# Лабораторная работа №1
## Тема: Формулирование требований к программной системе
## Цель работы: Научиться анализировать поставленную задачу, формулировать функциональные и нефункциональные требования к проектируемой системе.


## 1. Перечень заинтересованных лиц (стейкхолдеров)

1. **Аналитик данных**
    - Использует систему для загрузки данных, очистки, анализа, построения визуализаций и базовых моделей.
2. **ML-инженер**
    - Работает с сложными моделями, ансамблями, настройкой гиперпараметров, генерацией прогнозов.
3. **Архитектор данных**
    - Отвечает за структуры хранения, способы загрузки и форматы данных.
4. **Бизнес-заказчик / владелец продукта**
    - Заинтересован в получении аналитических выводов, прогнозов и отчётности.
5. **Конечный пользователь аналитической отчётности**
    - Получает выводы и визуализации, но сам не работает с инструментами анализа.

---

## 2. Перечень функциональных требований

1. Система должна предоставлять пользователю возможность загрузки данных из Хранилища Данных;
2. Система должна предоставлять пользователю возможность предобработки данных:
    - Нормализация;
    - Категоризация;
    - Фильтрация выбросов;
    - Очистка пропусков в значениях.
3. Система должна предоставлять пользователю возможность проводить аналитику:
    - Статистические показатели;
    - Корреляции;
    - Графики временных рядов.
4. Пользователь должен иметь возможность построить регрессионные модели;
5. Пользователь должен иметь возможность оценить модели по регрессионным метрикам;
6. Система должна предоставлять пользователю возможность визуализации и сохранения графиков.

## 4. Перечень сделанных предположений

1. Система используется одним специалистом на одном устройстве;
2. Датасеты не превышают нескольких сотен тысяч строк;
3. В RAW зоне хранятся сырые данные без изменений;
4. Пользователь знает базовые методы анализа данных;
5. Не требуется автоматическое документирование результатов.

---

## 5. Нефункциональные требования

1. **Производительность**
    - Обработка датасета до 200-300 тыс. строк должна занимать <= 30 секунд.

2. **Надёжность**
    - Система должна обрабатывать типичные ошибки данных с выводом понятного сообщения, без прерывания своей работы или аварийного завершения выполнения

3. **Удобство использования**
    - Логичная последовательность: загрузка --> обработка --> анализ --> модели --> итог.

4. **Расширяемость**
    - Возможность добавлять новые модели и методы анализа без изменения существующих модулей.
